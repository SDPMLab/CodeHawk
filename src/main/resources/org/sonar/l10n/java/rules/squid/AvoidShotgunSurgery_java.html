This rule detects if Shotgun Surgery has occurred in the code.
Shotgun surgery is when a small change in one part of the system causes the rest of the classes to change, and these classes are scattered throughout the rest of the system.
Shotgun Surgery detection strategy : "Operation is called by too many other methods (Changing Methods > 7)" and "Incoming calls are from many classes (Changing Classes > 10)"

Noncompliant Example

Compliant Solution

1.Move more responsibility to the classes defining Shotgun Surgery methods, from the client classes of these functions.
  Do this especially when the definition classes of the Shotgun Surgery methods is small and/or not complex and/or it is or has a tendency to become a Data Class. 
  Move Behavior Close to the Data presents step by step how to move behavior close to the data it uses and can be helpful here.

2.The Shotgun Surgery methods that are very large and complex (e.g., tending to become a Brain Method) should be especially analyzed and taken care of by the maintainers of the system (e.g., by increasing the number of test cases for the method, or by trying to refactor it). 
  We recommend this as such methods have a higher potential for change and/or malfunction potentially having a severe impact on the rest of the system. 
  Identifying clearly the stable interfaces in the system is a good way to reduce the candidates for refactorings.
